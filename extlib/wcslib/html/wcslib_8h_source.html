1962 @endverbatim
1963 In this example, if an error was generated in one of the prjset() functions,
1964 wcsperr() would print an error traceback starting with wcsset(), then
1965 celset(), and finally the particular projection-setting function that
1966 generated the error. For each of them it would print the status return value,
1967 function name, source file, line number, and an error message which may be
1968 more specific and informative than the general error messages reported in the
1969 first example. For example, in response to a deliberately generated error,
1970 the @c twcs test program, which tests wcserr among other things, produces a
1971 traceback similar to this:
1972 @verbatim
1973 ERROR 5 in wcsset() at line 1564 of file wcs.c:
1974  Invalid parameter value.
1975 ERROR 2 in celset() at line 196 of file cel.c:
1976  Invalid projection parameters.
1977 ERROR 2 in bonset() at line 5727 of file prj.c:
1978  Invalid parameters for Bonne's projection.
1979 @endverbatim
1980 
1981 Each of the @ref structs "structs" in @ref overview "WCSLIB" includes a
1982 pointer, called @a err, to a wcserr struct. When an error occurs, a struct is
1983 allocated and error information stored in it. The wcserr pointers and the
1984 @ref memory "memory" allocated for them are managed by the routines that
1985 manage the various structs such as wcsini() and wcsfree().
1986 
1987 wcserr messaging is an opt-in system enabled via wcserr_enable(), as in the
1988 example above. If enabled, when an error occurs it is the user's
1989 responsibility to free the memory allocated for the error message using
1990 wcsfree(), celfree(), prjfree(), etc. Failure to do so before the struct goes
1991 out of scope will result in memory leaks (if execution continues beyond the
1992 error).
