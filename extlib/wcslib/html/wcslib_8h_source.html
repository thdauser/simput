3349 In this example, if an error was generated in one of the prjset() functions,
3350 wcsperr() would print an error traceback starting with wcsset(), then
3351 celset(), and finally the particular projection-setting function that
3352 generated the error. For each of them it would print the status return value,
3353 function name, source file, line number, and an error message which may be
3354 more specific and informative than the general error messages reported in the
3355 first example. For example, in response to a deliberately generated error,
3356 the @c twcs test program, which tests wcserr among other things, produces a
3357 traceback similar to this:
3358 @verbatim
3359 ERROR 5 in wcsset() at line 1564 of file wcs.c:
3360  Invalid parameter value.
3361 ERROR 2 in celset() at line 196 of file cel.c:
3362  Invalid projection parameters.
3363 ERROR 2 in bonset() at line 5727 of file prj.c:
3364  Invalid parameters for Bonne's projection.
3365 @endverbatim
3366 
3367 Each of the @ref structs "structs" in @ref overview "WCSLIB" includes a
3368 pointer, called @a err, to a wcserr struct. When an error occurs, a struct is
3369 allocated and error information stored in it. The wcserr pointers and the
3370 @ref memory "memory" allocated for them are managed by the routines that
3371 manage the various structs such as wcsinit() and wcsfree().
3372 
3373 wcserr messaging is an opt-in system enabled via wcserr_enable(), as in the
3374 example above. If enabled, when an error occurs it is the user's
3375 responsibility to free the memory allocated for the error message using
3376 wcsfree(), celfree(), prjfree(), etc. Failure to do so before the struct goes
3377 out of scope will result in memory leaks (if execution continues beyond the
3378 error).
